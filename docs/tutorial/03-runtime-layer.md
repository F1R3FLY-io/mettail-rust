# Module 3: The Runtime Layer

**Goal**: Understand the traits and types that all generated code depends on.

The `runtime/` crate is the foundation. Every language generated by the `language!` macro depends on the types and traits defined here. It's small (~1200 lines total) and worth reading in full.

---

## 3.1 The `Term` Trait

**File**: `runtime/src/language.rs:94-107`

```rust
pub trait Term: fmt::Display + fmt::Debug + Send + Sync {
    fn clone_box(&self) -> Box<dyn Term>;
    fn term_id(&self) -> u64;
    fn term_eq(&self, other: &dyn Term) -> bool;
    fn as_any(&self) -> &dyn Any;
}
```

This is the most basic trait. Every AST node in every language implements it.

### Why These Methods?

**`clone_box(&self) -> Box<dyn Term>`** - You can't call `.clone()` on a `dyn Trait` directly (because `clone()` returns `Self`, and the compiler doesn't know the concrete type). So we have `clone_box()` which returns a new `Box<dyn Term>`.

**`term_id(&self) -> u64`** - A hash-based identifier for the term. Used to track terms through the rewrite graph. Two structurally equal terms will have the same `term_id`.

**`term_eq(&self, other: &dyn Term) -> bool`** - Structural equality check across trait objects. Uses `as_any()` internally to downcast and compare concrete types.

**`as_any(&self) -> &dyn Any`** - Enables downcasting from `dyn Term` back to a concrete type like `Proc`. This is the escape hatch from dynamic dispatch when you need the real type.

### The Trait Bounds

`Term: fmt::Display + fmt::Debug + Send + Sync` means:
- **Display** - Every term can be pretty-printed (the generated `Display` impl)
- **Debug** - Every term can be debug-printed
- **Send + Sync** - Terms can be sent between threads (needed for future parallelism)

## 3.2 The `Language` Trait

**File**: `runtime/src/language.rs:109-204`

This is the big one. The REPL interacts with languages exclusively through this trait.

```rust
pub trait Language: Send + Sync {
    // Identity
    fn name(&self) -> &'static str;
    fn metadata(&self) -> &'static dyn LanguageMetadata;

    // Parsing
    fn parse_term(&self, input: &str) -> Result<Box<dyn Term>, String>;
    fn parse_term_for_env(&self, input: &str) -> Result<Box<dyn Term>, String>;

    // Execution
    fn run_ascent(&self, term: &dyn Term) -> Result<AscentResults, String>;
    fn try_direct_eval(&self, term: &dyn Term) -> Option<Box<dyn Term>>;
    fn normalize_term(&self, term: &dyn Term) -> Box<dyn Term>;

    // Display
    fn format_term(&self, term: &dyn Term) -> String;

    // Environment
    fn create_env(&self) -> Box<dyn Any + Send + Sync>;
    fn add_to_env(&self, env: &mut dyn Any, name: &str, term: &dyn Term) -> Result<(), String>;
    fn remove_from_env(&self, env: &mut dyn Any, name: &str) -> Result<bool, String>;
    fn clear_env(&self, env: &mut dyn Any);
    fn substitute_env(&self, term: &dyn Term, env: &dyn Any) -> Result<Box<dyn Term>, String>;
    fn list_env(&self, env: &dyn Any) -> Vec<(String, String, Option<String>)>;
    // ... more env methods

    // Type inference
    fn infer_term_type(&self, term: &dyn Term) -> TermType;
    fn infer_var_types(&self, term: &dyn Term) -> Vec<VarTypeInfo>;
}
```

### Key Methods Explained

**`parse_term` vs `parse_term_for_env`**: Both parse strings into terms. The difference is that `parse_term` clears the variable cache first (for fresh evaluation), while `parse_term_for_env` preserves it (so environment bindings share variable identity with the terms they're substituted into).

**`run_ascent`**: The main execution entry point. Takes a term, runs the generated Ascent Datalog program on it, and returns all reachable terms, rewrites, and equivalence classes.

**`try_direct_eval`**: A fast path for purely native terms. If a term like `AddInt(NumLit(2), NumLit(3))` can be evaluated without Ascent, this returns the result directly. Falls back to `None` for terms with free variables or non-native constructors.

**`normalize_term`**: Beta-reduces lambda applications and flattens collections. This is called before Ascent execution so the Datalog engine doesn't have to deal with unreduced lambdas.

**`substitute_env`**: Replaces environment variable references in a term with their bound values. When you type `exec $name(fwd, x)` and `fwd` is in the environment, this method replaces `fwd` with its definition.

## 3.3 `AscentResults` - What Execution Produces

**File**: `runtime/src/language.rs:207-325`

```rust
pub struct AscentResults {
    pub all_terms: Vec<TermInfo>,                              // Every reachable term
    pub rewrites: Vec<Rewrite>,                                // Every rewrite edge
    pub equivalences: Vec<EquivClass>,                         // Equation-derived classes
    pub custom_relations: HashMap<String, RelationData>,       // User-defined Datalog relations
}
```

When you run `exec` in the REPL, you get back an `AscentResults`. Think of it as a **graph**:
- **Nodes** = `all_terms` (each term reachable from the initial term)
- **Directed edges** = `rewrites` (each rewrite from one term to another)
- **Undirected clusters** = `equivalences` (terms that are equal by equations)

### `TermInfo`

```rust
pub struct TermInfo {
    pub term_id: u64,        // Unique identifier (hash-based)
    pub display: String,     // Pretty-printed form
    pub is_normal_form: bool, // True if no rewrites apply to this term
}
```

### `Rewrite`

```rust
pub struct Rewrite {
    pub from_id: u64,              // Source term ID
    pub to_id: u64,                // Target term ID
    pub rule_name: Option<String>, // Which rule fired (e.g., "Comm", "Exec")
}
```

### Key Methods

```rust
impl AscentResults {
    fn normal_forms(&self) -> Vec<&TermInfo>;         // Terms with no outgoing rewrites
    fn rewrites_from(&self, term_id: u64) -> Vec<&Rewrite>;  // Rewrites from a specific term
    fn normal_form_reachable_from(&self, id: u64) -> Option<&TermInfo>;  // BFS to a normal form
}
```

**`normal_form_reachable_from`** is particularly useful - it does a BFS through the rewrite graph to find a normal form reachable from any given term.

## 3.4 `HashBag<T>` - Multisets for Parallel Composition

**File**: `runtime/src/hashbag.rs`

In rho-calculus, parallel composition `{ P | Q | R }` is:
- **Associative**: `{ P | { Q | R } }` = `{ { P | Q } | R }` = `{ P | Q | R }`
- **Commutative**: `{ P | Q }` = `{ Q | P }`
- Allows **duplicates**: `{ P | P }` is valid (two copies of P running)

This is exactly a **multiset** (bag). `HashBag<T>` implements it:

```rust
pub struct HashBag<T: Clone + Hash + Eq> {
    counts: HashMap<T, usize>,   // Element → count
    total_count: usize,          // Sum of all counts
}
```

### Key Properties

**Order-independent equality**: `{a, b, c}` == `{c, a, b}` because equality compares element counts, not order:

```rust
impl<T: Clone + Hash + Eq> PartialEq for HashBag<T> {
    fn eq(&self, other: &Self) -> bool {
        self.total_count == other.total_count && self.counts == other.counts
    }
}
```

**Deterministic hashing**: For use in Ascent relations (which need `Hash`), elements are sorted by their debug representation before hashing. This is slow but correct:

```rust
impl<T: Clone + Hash + Eq + Debug> Hash for HashBag<T> {
    fn hash<H: Hasher>(&self, state: &mut H) {
        self.total_count.hash(state);
        let mut entries: Vec<_> = self.counts.iter().collect();
        entries.sort_by_key(|(k, _)| format!("{:?}", k));
        for (elem, &count) in entries { ... }
    }
}
```

### Integration with Moniker

`HashBag` implements `BoundTerm<N>` from the moniker crate. This means variable binding operations (open, close, substitute) work correctly through collections:

```rust
impl<N, T> BoundTerm<N> for HashBag<T>
where T: Clone + Hash + Eq + BoundTerm<N> {
    fn close_term(&mut self, state: ScopeState, on_free: &impl OnFreeFn<N>) {
        // Rebuilds the map because closing may change element identity
        let old = std::mem::take(&mut self.counts);
        for (mut elem, count) in old {
            elem.close_term(state, on_free);
            self.counts.insert(elem, count);
        }
    }
    // ...
}
```

### Usage Example

```rust
let mut bag = HashBag::new();
bag.insert(Proc::PZero);
bag.insert(Proc::PZero);  // Two copies of PZero

assert_eq!(bag.len(), 2);
assert_eq!(bag.count(&Proc::PZero), 2);

// In the AST, this represents: { {} | {} }
// PPar(bag) is a parallel composition of two stopped processes
```

## 3.5 Variable Binding: `Scope`, `Binder`, and the Var Cache

**File**: `runtime/src/binding.rs`

Variable binding is one of the hardest problems in language implementation. Consider:

```
@(0)?x.{*(x)}    -- receive on @(0), bind to x, then deref x
@(0)?y.{*(y)}    -- this is the SAME program (alpha-equivalent)
```

The names `x` and `y` don't matter - what matters is the *binding structure*. The [moniker](https://github.com/brendanzab/moniker) crate handles this correctly.

### Core Types (from moniker)

```rust
// Re-exported from moniker:
pub use moniker::{Binder, BoundPattern, BoundTerm, BoundVar, FreeVar, Var};
```

- **`FreeVar<String>`** - A free variable (not bound by any binder). Has a `pretty_name` for display and a `unique_id` for identity.
- **`BoundVar`** - A bound variable (refers to a binder above it). Uses de Bruijn-like indices.
- **`Var<String>`** - Either `Var::Free(FreeVar)` or `Var::Bound(BoundVar)`.
- **`Binder<String>`** - The binding site. In `^x.p`, `x` is the binder.

### The `Scope` Wrapper

MeTTaIL wraps moniker's `Scope` to add `Hash` and `Ord`:

```rust
pub struct Scope<P, T> {
    inner: moniker::Scope<P, T>,
}
```

Why? Because Ascent relations need `Hash` for indexing, and term generation needs `Ord` for enumeration. Moniker's `Scope` doesn't implement these.

Key methods:
- **`Scope::new(pattern, body)`** - Creates a binding scope (closes the body)
- **`scope.unbind()`** - Opens the scope, returning `(pattern, body)` with fresh variable names
- **`scope.unsafe_pattern()` / `scope.unsafe_body()`** - Direct access without freshening (used in Ascent rules for stable identity)

### The Variable Cache

```rust
thread_local! {
    static VAR_CACHE: RefCell<HashMap<String, FreeVar<String>>> =
        RefCell::new(HashMap::new());
}
```

**Problem**: When parsing `@(0)?x.{*(x)}`, the two occurrences of `x` must produce the *same* `FreeVar`. Otherwise moniker won't recognize them as the same variable.

**Solution**: A thread-local cache. `get_or_create_var("x")` returns the same `FreeVar` every time within a parsing session.

**Critical**: `clear_var_cache()` must be called between independent parsing operations (between tests, between REPL `exec` calls that should have fresh variable contexts). Forgetting this causes subtle bugs where variables from different terms accidentally share identity.

```rust
// From rhocalc_tests.rs:
fn fresh() {
    mettail_runtime::clear_var_cache();
}

fn run(input: &str) -> AscentResults {
    fresh();  // ← Always clear before parsing a new test term
    let lang = RhoCalcLanguage;
    let term = lang.parse_term(input).unwrap();
    lang.run_ascent(term.as_ref()).unwrap()
}
```

### `OrdVar` Wrapper

```rust
pub struct OrdVar(pub Var<String>);
```

Wraps `Var<String>` to add `Ord` (total ordering). Moniker doesn't implement `Ord` for `Var`, but MeTTaIL needs it for sorted collections and deterministic output. The ordering compares:
- Free vs Bound (Free < Bound)
- Free vars by their unique ID (via hashing)
- Bound vars by their de Bruijn indices

## 3.6 `CanonicalFloat64` - Floats That Work in Datalog

**File**: `runtime/src/canonical_float.rs`

**Problem**: IEEE 754 floats break the rules that Datalog needs:
- `NaN != NaN` (violates reflexivity for `Eq`)
- Multiple bit patterns for `NaN` and `+0`/`-0`
- No total ordering (NaN is unordered)

**Solution**: `CanonicalFloat64` wraps `f64` and provides correct `Eq`, `Hash`, and `Ord`:

```rust
pub struct CanonicalFloat64(f64);
```

- Canonicalizes `NaN` to a single representation
- Treats `-0.0` as equal to `+0.0`
- Provides total ordering (NaN sorts to the end)
- Implements `Eq` and `Hash` consistently

This is used whenever a language has `![f64] as Float` in its type definitions.

## 3.7 `TermType` - Runtime Type Information

```rust
pub enum TermType {
    Base(String),                        // "Proc", "Name", "Int"
    Arrow(Box<TermType>, Box<TermType>), // [Name -> Proc]
    MultiArrow(Box<TermType>, Box<TermType>), // [Name* -> Proc]
    Unknown,
}
```

Used by the REPL's `typeof` command to display inferred types. The `Language` trait has methods:
- `infer_term_type(term)` - What type is this term?
- `infer_var_types(term)` - What types are the free variables?
