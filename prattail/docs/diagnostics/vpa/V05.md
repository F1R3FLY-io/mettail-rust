# V05: weighted-vpa-non-determinizable

**Severity:** Warning
**Module:** M4 Weighted VPA
**Feature gate:** `vpa`

## What It Detects

- A weighted visibly pushdown automaton has non-deterministic transitions that cannot be safely determinized due to alphabet mismatches between call and return symbols.
- Specifically, the VPA is flagged as non-determinizable (`!result.is_determinizable`) and has alphabet mismatches (`!result.alphabet_mismatches.is_empty()`). Determinization of such a VPA would require exponential state blowup, making it impractical for compile-time analysis.

## Example

### Triggering Code

```
language! {
    name: NestedLang;
    primary: Expr;

    category Expr {
        Paren  = "(" Expr ")";
        Brack  = "[" Expr "]";
        Curly  = "{" Expr "}";
        Angle  = "<" Expr ">";
        Add    = Expr "+" Expr;
        Lit    = <int>;
    }

    // Alphabet mismatch: "(" is a call symbol but also appears
    // in contexts where it doesn't have a matching return ")",
    // creating nondeterministic VPA transitions.
    category Type {
        TFun   = "(" Type ")" "->" Type;   // "(" as call
        TGroup = "(" Type ")";              // "(" as grouping (different return context)
    }
}
```

### Diagnostic Output

```
warning[V05] (NestedLang): weighted VPA is non-determinizable with 2 alphabet mismatches -- determinization may cause exponential blowup
  = hint: consider restricting call/return patterns to reduce nondeterminism
```

### Fix

```
// Use distinct delimiters for different nesting contexts:
category Type {
    TFun   = "fn" "(" Type ")" "->" Type;  // disambiguate with "fn" prefix
    TGroup = "(" Type ")";
}

// Or ensure each call symbol has exactly one return symbol partner:
// "(" always pairs with ")", "[" always pairs with "]", etc.
```

## Rationale

- VPA determinization uses weighted subset construction with stack: at call symbols, the entire weighted macro-state `HashMap<usize, W>` is pushed. At return symbols, the macro-state is popped and composed. Alphabet mismatches break this scheme because the same symbol acts as both call and non-call in different contexts.
- The Alur-Madhusudan determinization theorem guarantees that deterministic VPAs recognize exactly the visibly pushdown languages, but the construction is worst-case exponential in the number of states. Alphabet mismatches exacerbate this because they multiply the nondeterministic branching.
- Non-determinizable VPAs prevent several downstream analyses: weighted inclusion checking (V06), trampoline stack depth verification, and WPDS intersection.
- For `W: IdempotentSemiring`, weighted inclusion is decidable via product + emptiness. Non-determinizability makes the product construction intractable.
