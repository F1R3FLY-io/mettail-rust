# V06: weighted-vpa-inclusion-failure

**Severity:** Warning
**Module:** M4 Weighted VPA
**Feature gate:** `vpa`

## What It Detects

- A weighted VPA inclusion check is likely to fail or be prohibitively expensive: the VPA is non-determinizable and has more than 20 states.
- Inclusion checking determines whether the language of one VPA is contained in another (L(A) subseteq L(B)), which is essential for verifying that error recovery does not introduce inputs outside the intended language. When the VPA is large and non-determinizable, the product construction required for inclusion checking becomes intractable.

## Example

### Triggering Code

```
language! {
    name: RecoveryLang;
    primary: Stmt;

    category Stmt {
        Block = "{" StmtList "}";
        If    = "if" Expr "{" StmtList "}" "else" "{" StmtList "}";
        While = "while" Expr "{" StmtList "}";
        For   = "for" Binding "in" Expr "{" StmtList "}";
        Match = "match" Expr "{" ArmList "}";
        Try   = "try" "{" StmtList "}" "catch" "(" Pat ")" "{" StmtList "}";
        // ... 25+ states after VPA construction
    }

    // Recovery rules that widen acceptance:
    recovery {
        insert_close_brace;
        delete_unexpected;
        replace_delimiter;
    }
}
```

### Diagnostic Output

```
warning[V06] (RecoveryLang): weighted VPA inclusion check may fail -- 25 states and non-determinizable
  = hint: tighten recovery predicates or increase cost thresholds
```

### Fix

```
// Option 1: Reduce VPA state count by simplifying nesting structure.
// Option 2: Tighten recovery rules to reduce the recovery language:
recovery {
    insert_close_brace cost: 5.0;  // higher cost = less permissive
    delete_unexpected  cost: 8.0;
    // remove replace_delimiter -- too broad
}

// Option 3: Add explicit delimiter pairing to ensure determinizability.
```

## Rationale

- The lint fires when `!result.is_determinizable && result.state_count > 20`. The threshold of 20 states reflects the practical boundary where product construction begins to exhibit exponential blowup.
- Weighted VPA inclusion requires `W: IdempotentSemiring` (Theorem: weighted inclusion is decidable for idempotent semirings over VPLs). For non-idempotent weights, inclusion is undecidable in general.
- The inclusion check operates via product construction: build the product automaton `A x complement(B)` and test emptiness. When `B` is non-determinizable, complementation requires first determinizing `B`, which is the exponential bottleneck.
- Failed inclusion checks mean recovery correctness cannot be verified: the error recovery subsystem may accept inputs that violate the original grammar's structural constraints (e.g., unbalanced delimiters, mismatched nesting).
