# RT02: tautological-refinement-predicate

**Severity:** Note
**Module:** Refinement Type System
**Feature gate:** `type-system`

## What It Detects

A refinement type's predicate is trivially true for all values of the base type.
The refinement `{ x: T | P(x) }` is equivalent to the base type `T` — the
predicate adds no constraint.

Detection checks for tautological patterns: literal `true`, `x == x`,
`x >= MIN /\ x <= MAX` (full range), and predicates where the constraint theory
reports universal satisfiability.

## Example

### Triggering Code

```
types {
    Int:i64 { ... }
    AnyInt = { x: Int | true };
}
```

The predicate `true` is satisfied by all integers — `AnyInt` is identical to `Int`.

### Diagnostic Output

```
note[RT02] (MyLang): refinement type 'AnyInt' has tautological predicate: equivalent to base type 'Int'
  = hint: simplify to the base type 'Int' unless the refinement serves as documentation
```

### Fix

```
// Option 1: Add a meaningful predicate.
PosInt = { x: Int | x > 0 };

// Option 2: Use the base type directly.
// (Remove AnyInt and use Int wherever it appears.)
```

## Rationale

- A tautological refinement adds compile-time analysis cost (satisfiability
  checking, pairwise comparison) and runtime cost (an `is_refined_*` relation
  that contains the same tuples as the base category relation) for no benefit.
- This is typically a typo or incomplete definition that the author intended to
  constrain further.
- Severity is Note rather than Warning because the redundancy is harmless.
