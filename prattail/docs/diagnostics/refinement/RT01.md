# RT01: unsatisfiable-refinement-predicate

**Severity:** Warning
**Module:** Refinement Type System
**Feature gate:** `type-system`

## What It Detects

A refinement type's predicate is provably unsatisfiable — there exists no value
of the base type that satisfies the constraint. The refinement type `{ x: T | P(x) }`
is empty (uninhabited), making any use of this type dead code.

Detection uses `TheoryAlgebra::is_satisfiable()` for Presburger predicates
(automata-based emptiness), `LatticeTheory` for subtype constraints, and
heuristic analysis for behavioral predicates.

## Example

### Triggering Code

```
types {
    Int:i64 { ... }
    NeverInt = { x: Int | x > 0 /\ x < 0 };
}
```

The conjunction `x > 0 /\ x < 0` has no satisfying integer assignment.

### Diagnostic Output

```
warning[RT01] (MyLang): refinement type 'NeverInt' has unsatisfiable predicate: contradictory constraints
  = hint: check the predicate for contradictory or impossible constraints
```

### Fix

```
// Option 1: Fix the predicate to define a non-empty region.
SmallPosInt = { x: Int | x > 0 /\ x < 100 };

// Option 2: Remove the dead type entirely.
```

## Rationale

- An uninhabited refinement type wastes compile-time analysis resources and may
  indicate a logic error in the type definition. In process calculi, a channel
  typed with an uninhabited refinement type will never match any received value,
  causing silent deadlock.
- Detection is sound for Presburger (automata-based decision procedure) and
  lattice-based predicates (finite domain). For behavioral predicates involving
  arbitrary relations, the analysis is conservative — RT01 fires only when
  unsatisfiability can be proven.
