# RT04: refinement-subtype-detected

**Severity:** Note
**Module:** Refinement Type System
**Feature gate:** `type-system`

## What It Detects

One refinement type is provably a subtype of another — every value satisfying
the first predicate also satisfies the second. This means
`{ x: S | P(x) } <: { x: T | Q(x) }` where `S <: T` (base subtype) and
`∀x. P(x) ⟹ Q(x)` (predicate entailment).

Detection uses `TypeSystem::is_subtype()` for base types and
`TheoryAlgebra::implies()` for predicate entailment.

## Example

### Triggering Code

```
types {
    Int:i64 { ... }
    PosInt = { x: Int | x > 0 };
    SmallPosInt = { x: Int | x > 0 /\ x < 100 };
}
```

Every value in `SmallPosInt` (positive and less than 100) is also in `PosInt`
(positive). So `SmallPosInt <: PosInt`.

### Diagnostic Output

```
note[RT04] (MyLang): refinement type 'SmallPosInt' is a subtype of 'PosInt'
  = hint: this subtype relationship is available for dispatch optimization
```

### Fix

No fix needed — this is informational. The subtype relationship can be
leveraged for:

- **Dispatch optimization**: When dispatching on refinement types, `SmallPosInt`
  rules can be checked before `PosInt` rules without ambiguity.
- **Implicit coercion**: A value of type `SmallPosInt` can safely be used
  wherever `PosInt` is expected.

## Rationale

- Subtype relationships between refinement types are often intentional (defining
  a type hierarchy within a base type). This lint confirms the compiler's
  analysis agrees with the author's intent and makes the relationship explicit.
- For dispatch safety analysis, detected subtype pairs inform priority ordering:
  the more specific type (subtype) should be checked first.
