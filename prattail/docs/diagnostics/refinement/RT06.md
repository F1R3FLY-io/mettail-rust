# RT06: refinement-type-shadows-base

**Severity:** Warning
**Module:** Refinement Type System
**Feature gate:** `type-system`

## What It Detects

A refinement type's name is identical to (or collides with) one of the
language's base type category names. This shadows the base type, making it
impossible to refer to the unrefined base type by its original name.

Detection compares each `RefinementTypeDef::name` against
`LanguageDef::types[*].name` (case-insensitive).

## Example

### Triggering Code

```
types {
    Int:i64 { ... }
    Int = { x: Int | x > 0 };  // shadows base type 'Int'!
}
```

The refinement type `Int` has the same name as the base category `Int`. Any
subsequent reference to `Int` is ambiguous.

### Diagnostic Output

```
warning[RT06] (MyLang): refinement type 'Int' shadows base type 'Int'
  = hint: rename the refinement type to avoid ambiguity (e.g., 'PosInt')
```

### Fix

```
// Rename the refinement type to be distinct from the base type.
PosInt = { x: Int | x > 0 };
```

## Rationale

- Name shadowing creates ambiguity in type references throughout the grammar,
  pattern matching, and guard predicates. When a Comm rule references type `Int`,
  should it use the base category (all integers) or the refinement (positive
  integers only)?
- While some type systems allow shadowing (and resolve to the most specific
  meaning), PraTTaIL generates distinct Ascent relations for base types
  (`int(Int)`) and refinements (`is_refined_int(Int)`). Shadowing the name
  does not shadow the relation, so the behavior is confusing.
- This is always fixable by choosing a distinct name for the refinement type.
