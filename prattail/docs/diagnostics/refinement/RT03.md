# RT03: empty-refinement-intersection

**Severity:** Warning
**Module:** Refinement Type System
**Feature gate:** `type-system`

## What It Detects

Two refinement types that refine the same base type have provably empty
intersection — no value can satisfy both predicates simultaneously. This may
indicate an incomplete type partition or a logic error.

Detection performs pairwise analysis: for refinement types
`{ x: S | P(x) }` and `{ x: T | Q(x) }`, if `S` and `T` share a common
base type (or one is a subtype of the other), the pipeline checks whether
`P(x) /\ Q(x)` is satisfiable.

## Example

### Triggering Code

```
types {
    Int:i64 { ... }
    PosInt = { x: Int | x > 0 };
    NegInt = { x: Int | x < 0 };
}
```

`PosInt ∩ NegInt` is empty — no integer is both greater than 0 and less than 0.

### Diagnostic Output

```
warning[RT03] (MyLang): refinement types 'PosInt' and 'NegInt' have empty intersection
  = hint: these types are disjoint — verify this is intentional
```

### Fix

If disjointness is intentional (e.g., defining a type partition), no action
is needed. If unintentional:

```
// Option 1: Adjust predicates to allow overlap.
PosInt = { x: Int | x >= 0 };  // Now 0 is in both
NegInt = { x: Int | x <= 0 };
```

## Rationale

- When two refinement types over the same base are disjoint, a value can belong
  to at most one. If the grammar author expects some values to match both (e.g.,
  for overloaded dispatch), the empty intersection reveals a logic error.
- When disjointness is intentional, this lint serves as documentation that the
  types form a partition — the Note confirms the compiler's analysis agrees
  with the author's intent.
- The analysis is sound for Presburger predicates (automata intersection +
  emptiness) and conservative for behavioral predicates (heuristic complement
  detection).
