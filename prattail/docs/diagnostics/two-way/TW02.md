# TW02: one-way-sufficient

**Severity:** Note
**Module:** M11 Weighted Two-Way Transducers
**Feature gate:** `two-way-transducer`

## What It Detects

- A weighted two-way transducer has zero backward states (`Q_backward = emptyset`) and is determined to be one-way equivalent, meaning the bidirectional transduction capability is unused.
- The transducer can be replaced by a standard one-way WFST with lower complexity (no crossing sequence tracking needed), saving both compile-time and runtime overhead.

## Example

### Triggering Code

```
language! {
    name: SimpleTransform;
    primary: Token;

    category Token {
        Ident = <ident>;
        Num   = <int>;
        Op    = "+" | "-" | "*";
    }

    // Two-way transducer that only uses forward states:
    // It never moves the head backward.
    two_way_transform normalize {
        state S0 (forward);
        state S1 (forward);
        state S2 (forward);
        // All states are forward -- no backward movement
        transition(S0, Ident) -> (S1, [Ident], weight: 1.0);
        transition(S1, Op)    -> (S2, [Op],    weight: 1.0);
        transition(S2, Num)   -> (S0, [Num],   weight: 1.0);
    }
}
```

### Diagnostic Output

```
note[TW02] (SimpleTransform): two-way transducer (3 states, 0 backward) is one-way equivalent -- one-way transducer suffices
  = hint: use a standard one-way transducer for simpler implementation
```

### Fix

```
// Replace the two-way transducer with a standard one-way WFST:
transform normalize {
    state S0;
    state S1;
    state S2;
    transition(S0, Ident) -> (S1, [Ident], weight: 1.0);
    transition(S1, Op)    -> (S2, [Op],    weight: 1.0);
    transition(S2, Num)   -> (S0, [Num],   weight: 1.0);
}
```

## Rationale

- The lint fires when `result.is_one_way_equivalent && result.num_backward == 0`. A W2T with no backward states is trivially one-way: all transitions move the head right, so the transducer processes input left-to-right exactly like a standard WFST.
- The key overhead of two-way transducers is crossing sequence tracking: the `transduce(input)` simulation runs in O(|input|^2 x |Q|^2) for genuine two-way transducers, versus O(|input| x |Q|) for one-way. Eliminating the two-way machinery saves quadratic overhead.
- The `to_one_way()` operation (crossing sequence construction, Theorem 3.1) can convert any W2T to a one-way transducer, but at exponential cost O(2^(|Q|^2)) in the worst case. When the W2T is already one-way, no conversion is needed -- it is directly usable as a WFST.
- In the `transducer.rs` integration, a one-way-equivalent W2T can be wrapped as a standard `OptimizationPass` rather than a `TwoWayOptimizationPass`, enabling integration with the existing single-pass pipeline.
