# PB02: tautological-arithmetic-guard

**Severity:** Warning
**Module:** Presburger Arithmetic
**Feature gate:** `presburger`

## What It Detects

An arithmetic guard is always satisfied -- the Presburger automata NFA accepts all
valid inputs. Every integer assignment to the guard's free variables satisfies the
constraint, making the guard redundant. Removing it has no effect on runtime
behavior but eliminates unnecessary compile-time and runtime evaluation overhead.

Detection works by complementing the guard's NFA and checking the complement for
emptiness. If the complement is empty, the original NFA is universal (accepts
everything), confirming the guard is tautological.

## Example

### Triggering Code

```
language! {
    name: Dispatch;
    primary: Proc;

    category Proc {
        Always = "for" "(" "@x" ":" "x >= 0 OR x < 0" "<-" ch ")" "{" Proc "}";
        Zero = "0";
    }
}
```

The guard `x >= 0 OR x < 0` covers the entire integer domain -- every integer is
either non-negative or negative.

### Diagnostic Output

```
warning[PB02] (Dispatch): arithmetic guard 'x >= 0 OR x < 0' is tautological (redundant)
  = note: guard accepts all valid inputs and can be removed
```

### Fix

```
// Remove the redundant guard -- the receive already accepts all values.
Always = "for" "(" "@x" "<-" ch ")" "{" Proc "}";
```

## Rationale

- Tautological guards consume compilation resources (NFA construction, product
  operations) and runtime resources (guard evaluation per message) without
  providing any filtering benefit.
- The universality check (complement emptiness) runs in time linear in the
  complement NFA state count. For quantifier-free formulas this is efficient;
  quantified formulas may involve an exponential blowup from complementation but
  this is inherent to the Presburger decision procedure.
- Unlike SYM02 (overlap), a tautological guard does not cause ambiguity -- it
  simply adds no discriminating power. This is informational rather than a
  correctness concern.
