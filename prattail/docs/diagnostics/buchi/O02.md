# O02: weighted-buchi-heavy-cycle

**Severity:** Note
**Module:** M2 Weighted Buchi Automata
**Feature gate:** `omega`

## What It Detects

- A weighted Buchi automaton has an accepting cycle where more than half of all states are accepting states.
- This suggests the automaton is overly permissive: it accepts most infinite behaviors, which weakens the discriminating power of the liveness property it encodes.

## Example

### Triggering Code

```
language! {
    name: Monitor;
    primary: Event;

    category Event {
        Request  = "req" <ident>;
        Response = "resp" <ident>;
        Timeout  = "timeout";
        Ack      = "ack";
    }

    // Liveness property with too many accepting states:
    // 4 out of 6 states are accepting, making the property
    // almost trivially satisfied.
    liveness responsive: eventually(Response | Ack | Timeout | Request);
}
```

### Diagnostic Output

```
note[O02] (Monitor): accepting cycle weight is high (4/6 states are accepting)
  = hint: consider whether all accepting states are intentional
```

### Fix

```
// Tighten the liveness property to fewer accepting states:
liveness responsive: eventually(Response);

// Or split into separate, more targeted properties:
liveness has_response: eventually(Response);
liveness has_ack: eventually(Ack);
```

## Rationale

- The lint fires when `result.has_accepting_cycle && result.num_accepting > result.num_states / 2`. A majority of accepting states means the Buchi condition `Inf(rho) intersection F != emptyset` is easily satisfied -- almost any infinite run will hit an accepting state.
- In weighted Buchi automata, heavy accepting cycles accumulate large weight via `matrix_star()` over the cycle's SCC. This can cause numerical instability in semirings like `TropicalWeight` (overflow) or `LogWeight` (underflow).
- A liveness property where most states are accepting provides weak guarantees: it says "eventually something happens" where "something" is almost anything. More selective accepting sets yield stronger, more useful properties.
- This is an informational note rather than a warning because the pattern is not necessarily wrong -- some specifications intentionally accept broad classes of behavior.
