# PR01: low-selectivity-rule

**Severity:** Warning
**Module:** M7 Probabilistic Automata
**Feature gate:** `probabilistic`

## What It Detects

- A grammar rule handles less than 1% of expected inputs based on the probabilistic model trained from corpus data.
- Low-selectivity rules are effectively dead code in practice: they match so infrequently that the compile-time cost of generating their parsing logic may outweigh their runtime benefit.

## Example

### Triggering Code

```
language! {
    name: Calculator;
    primary: Expr;

    category Expr {
        Add = Expr "+" Term;
        Sub = Expr "-" Term;
        Mul = Term;
    }

    category Term {
        Num    = <int>;
        Paren  = "(" Expr ")";
        Hyper  = "hyper" "(" Expr "," Expr "," Expr ")";  // < 1% of inputs
    }
}
```

If corpus analysis shows `Hyper` is used in less than 1% of parsed expressions, it is flagged.

### Diagnostic Output

```
warning[PR01] (Calculator): rule 'Hyper' handles <1% of expected inputs
  = hint: consider removing or specializing this low-frequency rule
```

### Fix

```
// Option 1: Move the rare rule behind a feature gate or extension module.
// Option 2: Keep it but accept the warning -- rare rules may still be
//   needed for correctness.
// Option 3: If the rule is truly unused, remove it.

// Option 4: Increase the rule's weight to reflect intentional inclusion:
category Term {
    Num    = <int>;
    Paren  = "(" Expr ")";
    #[allow(low_selectivity)]
    Hyper  = "hyper" "(" Expr "," Expr "," Expr ")";
}
```

## Rationale

- Selectivity is computed via the forward algorithm over the probabilistic automaton using `LogWeight`. A rule's selectivity is the fraction of total language probability routed through its transitions.
- The 1% threshold is based on corpus-trained weights: `train_from_corpus()` uses Baum-Welch EM via `forward_backward.rs` to learn transition probabilities from example programs.
- Low-selectivity rules increase parsing overhead because the parser must check their FIRST sets during dispatch, even though they almost never match. In prediction-driven parsers, this wastes branch prediction resources.
- In `cost_benefit.rs`, low-selectivity rules have high cost-to-benefit ratios: their optimization passes fire rarely, so the fixed cost of generating the optimization logic is amortized over very few executions.
