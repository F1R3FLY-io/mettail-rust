# EG01: discovered-equivalence

**Severity:** Note
**Module:** E-Graph Equality Saturation
**Feature gate:** `egraph`

## What It Detects

The e-graph equality saturation phase discovered that two distinct grammar
terms (constructors, guard expressions, or rewrite targets) are semantically
equivalent under the grammar's declared equations and rewrite rules. The two
terms belong to the same e-class after saturation, meaning they can be used
interchangeably without affecting semantics.

Detection works by inserting all grammar-derived terms into the e-graph,
running equality saturation with the grammar's equation and rewrite rule set,
and then scanning for e-classes that contain e-nodes originating from
syntactically distinct source-level terms. Each such e-class witnesses a
non-trivial equivalence that the grammar author may not have explicitly
intended.

This diagnostic is informational: it does not indicate a problem, but surfaces
equivalences that may reveal:

- Redundant constructors that could be consolidated.
- Unintended semantic collapse where distinct syntax maps to the same meaning.
- Optimization opportunities where one equivalent form is cheaper to evaluate
  or generates smaller code.

## Example

### Triggering Code

```rust
language! {
    name: BoolSimplify;
    primary: Expr;

    category Expr {
        True  = "true";
        False = "false";
        Not   = "!" Expr;
        And   = Expr "&&" Expr;
        Or    = Expr "||" Expr;
    }

    equations {
        // Double negation elimination
        Not(Not(x)) = x;
        // De Morgan's law
        Not(And(a, b)) = Or(Not(a), Not(b));
        Not(Or(a, b))  = And(Not(a), Not(b));
    }
}
```

After equality saturation, the e-graph discovers that `Not(Not(Not(x)))` and
`Not(x)` are equivalent (via two applications of the double-negation rule).
This is a derived equivalence not explicitly stated in the equations but
logically implied by them.

### Diagnostic Output

```
note[EG01] (BoolSimplify): discovered equivalence: `Not(Not(Not(x)))` = `Not(x)` (e-class #42, 2 members)
  = hint: consider whether both forms are needed, or consolidate to the simpler one
```

### Response

No action is required. This diagnostic is informational. Possible responses:

1. **Consolidate**: if one form is clearly simpler (fewer nodes, lower depth),
   adjust the grammar or rewrite rules so only the simpler form is produced.

2. **Acknowledge**: if both forms serve distinct syntactic purposes (e.g.,
   different source representations that normalize to the same thing), no
   change is needed.

3. **Investigate**: if the equivalence is unexpected, it may reveal an
   over-general equation that collapses more terms than intended. Review the
   equation set for unintended interactions.

## Rationale

- Surfacing implicit equivalences helps grammar authors understand the full
  consequences of their equation sets. A small set of equations can produce a
  combinatorially large set of derived equivalences, and some of these may be
  surprising or unintended.

- Redundant constructors that are equivalent under the equations waste codegen
  resources: the parser generates distinct match arms, evaluation paths, and
  display implementations for terms that are semantically identical. Identifying
  these enables consolidation.

- For hash-consing-based term representations (used in PraTTaIL's Ascent
  codegen), equivalent terms that are not hash-consed to the same representative
  waste memory and slow equivalence checks. EG01 identifies candidates for
  hash-consing unification.
