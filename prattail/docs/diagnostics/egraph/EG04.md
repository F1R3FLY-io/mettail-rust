# EG04: joinability-witness

**Severity:** Note
**Module:** E-Graph Equality Saturation
**Feature gate:** `egraph`

## What It Detects

The e-graph equality saturation phase found a **joinability witness** for a
critical pair that was initially flagged as non-joinable by the TRS confluence
analysis (T01). After inserting both sides of the critical pair into the
e-graph and running equality saturation, the two terms ended up in the same
e-class, proving that they are joinable -- there exists a common reduct that
both terms can reach via the grammar's rewrite rules.

Detection works as follows:

1. The TRS confluence analysis (T01) identifies critical pairs (s, t) where
   overlapping rewrite rules produce divergent reducts.
2. For each non-joinable critical pair, the e-graph inserts both s and t.
3. Equality saturation applies all rewrite rules simultaneously, exploring all
   possible rewriting sequences within the resource limits.
4. After saturation, if `find(s) == find(t)` (both terms are in the same
   e-class), the pair is joinable. The e-graph extracts the common reduct w
   as the minimum-cost term in the shared e-class, providing a concrete
   **witness**: s ->* w and t ->* w.

When EG04 fires, the corresponding T01 diagnostic is suppressed or downgraded,
since the critical pair is not a genuine confluence failure -- the TRS analysis
simply could not find the joining path with its bounded sequential rewriting,
but the e-graph's parallel exploration succeeded.

## Example

### Triggering Code

```rust
language! {
    name: ArithNorm;
    primary: Expr;

    category Expr {
        Num  = /[0-9]+/;
        Var  = /[a-z][a-zA-Z0-9_]*/;
        Add  = Expr "+" Expr;
        Mul  = Expr "*" Expr;
        Neg  = "-" Expr;
    }

    equations {
        // Rule 0: distribute negation over addition
        Neg(Add(a, b)) = Add(Neg(a), Neg(b));
        // Rule 1: double negation elimination
        Neg(Neg(x)) = x;
    }
}
```

The term `Neg(Neg(Add(a, b)))` creates a critical pair between rules 0 and 1:

- **Rule 1 first**: `Neg(Neg(Add(a, b)))` -> `Add(a, b)`
- **Rule 0 on inner**: `Neg(Add(a, b))` -> `Add(Neg(a), Neg(b))`, then outer
  Neg: `Neg(Add(Neg(a), Neg(b)))` -> `Add(Neg(Neg(a)), Neg(Neg(b)))`

The bounded TRS analysis may not find the joining path from
`Add(Neg(Neg(a)), Neg(Neg(b)))` back to `Add(a, b)` if its reduction depth is
insufficient. The e-graph, however, applies rule 1 to both `Neg(Neg(a))` and
`Neg(Neg(b))` simultaneously, discovering that the two reducts are equivalent
to `Add(a, b)`.

### Diagnostic Output

```
note[EG04] (ArithNorm): joinability witness found for critical pair (rules 0, 1): `Add(Neg(Neg(a)), Neg(Neg(b)))` =* `Add(a, b)` (common reduct in e-class #17)
  = note: T01 suppressed for this critical pair
  = hint: the rewriting system is confluent for this overlap
```

### Response

No action is required. This diagnostic confirms that a potential confluence
failure (T01) is actually a false alarm -- the critical pair is joinable, and
the e-graph provides the concrete witness. Possible responses:

1. **No action**: the system is confluent for this critical pair. The e-graph
   has proven it automatically.

2. **Add an explicit rule**: if the joining path is long or complex, consider
   adding an explicit rewrite rule that shortcuts the path. This makes the
   confluence obvious to the TRS analysis without needing the e-graph, and may
   improve rewriting performance at runtime.

3. **Review the witness**: the common reduct reported in the diagnostic can be
   inspected to understand how the two divergent paths reconverge. This may
   reveal opportunities to simplify the rule set.

## Rationale

- The e-graph's parallel exploration of all rewriting sequences is strictly
  more powerful than bounded sequential rewriting for joinability checking. Many
  critical pairs that appear non-joinable under bounded depth are actually
  joinable -- they just require applying rules in a non-obvious order or at
  multiple positions simultaneously.

- By providing a concrete witness (the common reduct), the diagnostic gives
  the grammar author actionable information: not just "the pair is joinable"
  but "here is the term they both reduce to." This aids understanding and
  enables verification.

- Suppressing T01 when EG04 fires reduces diagnostic noise. Without the
  e-graph, T01 would report false positives for joinable-but-hard-to-find
  critical pairs, which could lead grammar authors to add unnecessary rules or
  restructure their grammars to avoid the (non-existent) confluence failure.

- The Note severity reflects that EG04 is a positive result: it confirms
  confluence, not a problem. It is emitted for informational purposes so the
  grammar author knows the e-graph contributed to the analysis.

## Interaction with EG03

If EG03 (saturation non-convergence) also fires, the e-graph may have failed
to find joinability witnesses for some critical pairs. In this case:

- Critical pairs where `find(s) == find(t)` despite non-convergence are still
  valid witnesses (EG04 fires for these).
- Critical pairs where `find(s) != find(t)` are inconclusive: the pair might
  be joinable via a path the e-graph did not have resources to explore. T01
  remains active for these pairs, but should be interpreted with the caveat
  that the e-graph analysis was incomplete.
