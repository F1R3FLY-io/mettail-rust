# EG02: simplifiable-guard

**Severity:** Note
**Module:** E-Graph Equality Saturation
**Feature gate:** `egraph`

## What It Detects

A guard expression attached to a grammar rule can be simplified to a
semantically equivalent but smaller or shallower expression, as discovered by
e-graph equality saturation. The e-graph found that the guard's e-class
contains an alternative e-node with strictly lower extraction cost (under the
configured cost function) than the original guard expression.

Detection proceeds as follows:

1. The guard expression is inserted into the e-graph as a term.
2. Equality saturation runs using the grammar's equations and rewrite rules.
3. After saturation, the extraction pass selects the minimum-cost
   representative from the guard's e-class.
4. If the extracted term differs from the original guard expression (i.e., the
   minimum-cost representative is not the original), EG02 fires.

This diagnostic is informational: the guard's semantics are unchanged, but the
simpler form may reduce codegen complexity, improve readability, or enable
further downstream optimizations (e.g., better SFA guard compilation, more
precise predicate dispatch).

## Example

### Triggering Code

```rust
language! {
    name: ArithOpt;
    primary: Expr;

    category Expr {
        Num   = /[0-9]+/;
        Var   = /[a-z][a-zA-Z0-9_]*/;
        Add   = Expr "+" Expr;
        Mul   = Expr "*" Expr;
        Neg   = "-" Expr;
    }

    equations {
        Neg(Neg(x)) = x;           // double negation
        Add(x, Num(0)) = x;        // additive identity
        Mul(x, Num(1)) = x;        // multiplicative identity
    }

    category Stmt {
        Assign = Var "=" Expr;
    }

    rewrites {
        // Guard: only fire when the expression is a double-negated addition
        // of zero. This is equivalent to just checking for the inner expression.
        Assign(v, Neg(Neg(Add(e, Num(0))))) => Assign(v, e);
    }
}
```

The guard pattern `Neg(Neg(Add(e, Num(0))))` is equivalent to `e` under the
declared equations (double negation + additive identity). The e-graph discovers
that the guard can be simplified, though the rewrite rule itself may still need
the full pattern for structural matching purposes.

More practically, consider a guard predicate:

```
guard(x > 0 /\ x > 0)
```

If the equation set includes idempotency of conjunction (`P /\ P = P`), the
e-graph simplifies this to `guard(x > 0)`.

### Diagnostic Output

```
note[EG02] (ArithOpt): guard `Neg(Neg(Add(e, Num(0))))` simplifiable to `e` via equality saturation (cost: 5 -> 1)
  = hint: consider replacing the guard with the simpler equivalent form
```

### Response

1. **Simplify the guard**: replace the guard expression with the simpler form
   suggested by the diagnostic. This reduces pattern-matching complexity in
   generated code and may improve parse-time performance.

2. **Keep the original**: if the guard's syntactic form carries documentation
   value (making the rule's intent clearer to readers), retain the original.
   The generated code may be slightly larger but correctness is unaffected.

3. **Automate**: if the cost-benefit analyzer determines that guard
   simplification produces measurable codegen improvements, a future
   optimization pass could apply these simplifications automatically.

## Rationale

- Simpler guard expressions produce smaller generated match arms. Each guard
  compiles to a pattern-matching prefix in the recursive descent parser; fewer
  nodes in the guard pattern means fewer comparisons at parse time.

- Guard simplification can enable further optimizations: a guard simplified to
  a single variable or constant may allow the predicate dispatch system to
  eliminate the guard check entirely (the rule always applies when reached).

- The cost delta reported in the diagnostic (`cost: X -> Y`) quantifies the
  improvement, allowing grammar authors to prioritize which guards are worth
  simplifying manually.

- For SFA-compiled guards (feature `symbolic-automata`), simpler guards produce
  smaller symbolic automata with fewer states and transitions, reducing both
  compilation time and runtime dispatch overhead.
