# EG03: saturation-non-convergence

**Severity:** Warning
**Module:** E-Graph Equality Saturation
**Feature gate:** `egraph`

## What It Detects

The e-graph equality saturation phase did not reach a fixed point within the
configured resource limits. New equivalences were still being discovered when
the analysis was forced to stop due to exceeding `max_nodes`, `max_iterations`,
or `time_limit_ms`. The results of the analysis are sound but potentially
**incomplete**: some valid equivalences may not have been discovered.

Non-convergence means the e-graph was still growing (new e-nodes being added,
new e-classes being merged) at termination. This can happen when:

1. **Rewrite rules are non-terminating**: the rule set generates an unbounded
   number of distinct terms. For example, a rule `x -> f(x)` generates
   `x, f(x), f(f(x)), ...` infinitely.

2. **Combinatorial explosion**: the interaction of multiple rules produces a
   super-linear number of new terms per iteration. Even terminating rule sets
   can require an impractically large e-graph to saturate.

3. **Resource limits are too low**: the grammar is complex enough that the
   default limits are insufficient for full saturation. Increasing the limits
   may resolve the warning.

This diagnostic is a warning because incomplete saturation can cause downstream
analyses to miss equivalences:

- EG01 may fail to report valid equivalences.
- EG02 may fail to simplify guards that have cheaper equivalent forms.
- EG04 may fail to find joinability witnesses for critical pairs that are
  actually joinable, causing T01 to fire as a false positive.

## Example

### Triggering Code

```rust
language! {
    name: ListOps;
    primary: Expr;

    category Expr {
        Nil     = "nil";
        Cons    = Expr "::" Expr;
        Append  = Expr "++" Expr;
        Rev     = "rev" "(" Expr ")";
        Map     = "map" "(" Expr "," Expr ")";
    }

    equations {
        // Append associativity
        Append(Append(x, y), z) = Append(x, Append(y, z));
        // Reverse distributes over append
        Rev(Append(x, y)) = Append(Rev(y), Rev(x));
        // Double reverse
        Rev(Rev(x)) = x;
        // Map distributes over append
        Map(f, Append(x, y)) = Append(Map(f, x), Map(f, y));
        // Map distributes over cons
        Map(f, Cons(x, xs)) = Cons(Map(f, x), Map(f, xs));
    }
}
```

The combination of associativity, reverse distribution, and map distribution
creates a combinatorial explosion: each application of one rule enables
multiple applications of others. For a term like
`Map(f, Rev(Append(a, Append(b, c))))`, saturation generates hundreds of
equivalent forms before the resource limit is reached.

### Diagnostic Output

```
warning[EG03] (ListOps): equality saturation did not converge (10,000 e-nodes, 30 iterations exhausted, 847 new merges in final iteration)
  = hint: increase `max_nodes` or `max_iterations`, or simplify the equation set
  = hint: set PRATTAIL_EGRAPH_MAX_NODES=50000 or PRATTAIL_EGRAPH_MAX_ITER=100
```

### Response

1. **Increase resource limits**: set `PRATTAIL_EGRAPH_MAX_NODES` and/or
   `PRATTAIL_EGRAPH_MAX_ITER` to higher values. Monitor compile time to
   ensure the increase is acceptable.

2. **Simplify the equation set**: review whether all equations are necessary.
   Associativity and distributivity rules are common sources of combinatorial
   explosion. Consider orienting some equations as one-directional rewrites
   rather than bidirectional equations, reducing the number of new terms
   generated per iteration.

3. **Restrict rule applicability**: add conditions or guards to limit which
   terms the rules apply to. For example, restricting `Map` distribution to
   terms of bounded depth prevents unbounded expansion.

4. **Accept incomplete results**: if the partially-saturated e-graph still
   provides useful results (some EG01/EG02/EG04 diagnostics fire before the
   limit is reached), the warning can be acknowledged without increasing
   limits. The results obtained are sound; only completeness is affected.

5. **Disable e-graph analysis**: for grammars where the equation set is known
   to be non-terminating and e-graph analysis provides limited value, disable
   the `egraph` feature gate entirely.

## Rationale

- Soundness is preserved: every equivalence reported by the e-graph before
  termination is valid. The warning is about **completeness**, not correctness.

- The diagnostic reports quantitative metrics (e-node count, iteration count,
  final-iteration merge count) to help diagnose the cause. A high
  final-iteration merge count indicates the e-graph is far from convergence; a
  low count suggests it is nearly saturated and a small limit increase may
  suffice.

- Non-convergence interacts with confluence analysis: if the e-graph cannot
  fully explore the rewriting space, some joinable critical pairs may appear
  non-joinable (T01 false positives). The grammar author should interpret T01
  results with caution when EG03 also fires.

- The warning severity (rather than note or error) reflects the practical
  impact: incomplete saturation degrades analysis quality but does not affect
  the correctness of generated parsers. The grammar will still compile and
  parse correctly; only the diagnostic and optimization quality is reduced.
