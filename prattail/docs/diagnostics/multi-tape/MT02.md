# MT02: multi-tape-disconnected

**Severity:** Note
**Module:** M8 Weighted Multi-Tape Automata
**Feature gate:** `multi-tape`

## What It Detects

- A tape in the multi-tape automaton has no `auto_intersect()` constraints linking it to any other tape.
- A disconnected tape operates independently: its acceptance is not correlated with any other tape's behavior. This means the multi-tape framework adds no value for this tape -- it could be analyzed as a standalone single-tape automaton with lower overhead.

## Example

### Triggering Code

```
language! {
    name: ThreeStream;
    primary: Proc;

    category Proc {
        Sync = "for" "(" "@x" "<-" ch1 "," "@y" "<-" ch2 "," "@z" "<-" ch3 ")" "{" Body "}";
    }

    multi_tape_constraint {
        auto_intersect(0, 1);  // ch1 and ch2 are correlated
        // ch3 (tape 2) has no constraints -- it is disconnected
    }
}
```

### Diagnostic Output

```
note[MT02] (ThreeStream): tape 2 has no auto-intersection constraints (independent channel)
  = hint: the disconnected tape can be analyzed independently
```

### Fix

```
// Option 1: Add a constraint linking the tape to others:
multi_tape_constraint {
    auto_intersect(0, 1);
    auto_intersect(1, 2);  // now tape 2 is correlated with tape 1
}

// Option 2: Analyze the disconnected tape separately as a single-tape automaton:
// Move ch3 out of the multi-channel receive:
category Proc {
    Sync2  = "for" "(" "@x" "<-" ch1 "," "@y" "<-" ch2 ")" "{" Body1 "}";
    Single = "for" "(" "@z" "<-" ch3 ")" "{" Body2 "}";
}
```

## Rationale

- The lint iterates over `result.disconnected_tapes`, a list of tape indices that appear in no `auto_intersect(i, j)` pair. A tape index `k` is disconnected if for all `(i, j)` pairs, `k != i && k != j`.
- Multi-tape automaton operations have complexity that scales with the product of tape sizes. A disconnected tape adds to the product without contributing any cross-tape constraints, inflating the state space unnecessarily.
- In multi-channel receives, a disconnected channel means the receive's synchronization semantics are partial: channels ch1 and ch2 may be correlated, but ch3 fires independently. This might be intentional (an independent listener) or accidental (a missing constraint).
- The `project(tape_idx)` operation extracts the single-tape behavior in O(|E|). Projecting a disconnected tape and analyzing it independently is always faster than including it in the multi-tape product.
