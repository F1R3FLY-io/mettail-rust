# UN03: subsumed-unification-guard

**Severity:** Note
**Module:** Unification
**Feature gate:** `unification`

## What It Detects

One structural pattern guard is strictly more general than another -- any input
term matching the specific pattern also matches the general one. Formally, pattern
A subsumes pattern B if there exists a substitution sigma such that A*sigma* = B
but no substitution tau makes B*tau* = A. The subsumed guard (B) is redundant when
both appear on the same channel: it can never fire exclusively.

Detection computes the MGU of A and B. If the MGU is a one-way match (A matches B
but B does not match A), subsumption holds.

## Example

### Triggering Code

```
language! {
    name: Matcher;
    primary: Proc;

    category Proc {
        // f(x, a) subsumes f(b, a): setting x = b in the general pattern yields the specific one.
        General  = "for" "(" "@t" ":" "f(x, a) = t" "<-" ch ")" "{" GenProc  "}";
        Specific = "for" "(" "@t" ":" "f(b, a) = t" "<-" ch ")" "{" SpecProc "}";
        Zero = "0";
    }
}
```

The pattern `f(x, a)` subsumes `f(b, a)`: the variable `x` in the general pattern
can be instantiated to `b`, yielding the specific pattern. Any term matching
`f(b, a)` also matches `f(x, a)`.

### Diagnostic Output

```
note[UN03] (Matcher): unification guard 'f(b, a)' is subsumed by 'f(x, a)' (redundant)
  = hint: the more general pattern already covers this case
```

### Fix

```
// Option 1: Remove the subsumed guard; handle all matching terms in one branch.
General = "for" "(" "@t" ":" "f(x, a) = t" "<-" ch ")" "{" GenProc "}";

// Option 2: Make guards disjoint by adding a constraint that excludes the general case.
General  = "for" "(" "@t" ":" "f(x, a) = t AND x != b" "<-" ch ")" "{" GenProc  "}";
Specific = "for" "(" "@t" ":" "f(b, a) = t"             "<-" ch ")" "{" SpecProc "}";
```

## Rationale

- Subsumption in first-order terms is decidable in linear time via one-way
  matching (a restricted form of unification where only one side's variables are
  instantiated).
- Subsumed guards waste dispatch effort: the runtime evaluates both patterns and
  must resolve the overlap. Under priority-ordered dispatch, the subsumed guard is
  effectively dead code. Under nondeterministic dispatch, it creates unnecessary
  branching.
- This is the structural analogue of PB03 (subsumed arithmetic guard) and SYM03
  (subsumed symbolic guard). All three detect the same logical condition --
  set-inclusion of solution spaces -- through different decision procedures adapted
  to their respective constraint domains.
