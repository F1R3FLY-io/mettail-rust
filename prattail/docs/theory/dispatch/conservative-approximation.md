# Conservative Approximation — Soundness of Two-Tier Dispatch

## Overview

PraTTaIL's module dispatch determines which of the 11 advanced automata modules
(M1--M11) must be activated for a given grammar. The dispatch operates at two
tiers with distinct information sources and precision levels. This document
provides a formal soundness argument that the two-tier architecture never
under-activates modules (no false negatives), while accepting that it may
over-activate them (false positives are harmless).

## 1. Two-Tier Architecture

### Definitions

**Definition 1.1** (Module Need). For a grammar G with predicate formulas
Φ = {φ₁, ..., φₙ}, the **actual module need** is:

  need(G) = { Mᵢ : some analysis task in G requires the capabilities of Mᵢ }

This set is generally undecidable to compute exactly (it would require computing
syntactic monoids for the predicate languages, which is exponential in the worst
case). Instead, PraTTaIL computes two conservative approximations.

**Definition 1.2** (Tier 1 -- Grammar-Structure Heuristics). The **grammar
heuristic signature** σ_β is computed by `classify_grammar()`, which examines
only the syntactic structure of grammar rules (category reference graphs, rule
counts, terminal symbols, non-terminal arities) without inspecting predicate
formulas. Formally:

  σ_β = classify_grammar(all_syntax, categories).aggregate_signature

**Definition 1.3** (Tier 2 -- Predicate-Level Feature Extraction). The
**predicate signature** σ_α is computed by `extract_features()` (or
`extract_features_mso()` for MSO formulas), which performs an O(|AST|)
post-order traversal of each predicate formula's AST, accumulating module bits
based on morphological pattern matching. Formally:

  σ_α = ⋃_{φ ∈ Φ} extract_features(φ, Γ).signature

where Γ is the channel context.

**Definition 1.4** (Dispatch Composition). The final dispatch signature is:

  σ_final = σ_β ∪ σ_α

When predicate expressions are available (i.e. the predicated-types codegen
pipeline has parsed them), σ_α provides exact variety classification per
Lemma 1.1 of `variety-classification.md`. When predicates are not yet available,
σ_β provides a structural over-approximation. When both are available, the union
ensures maximal coverage:

  σ_final = σ_β ∪ σ_α ⊇ max(σ_α, σ_β) ⊇ need(G)

The union is monotone (only sets bits, never clears) and idempotent
(σ ∪ σ = σ), so repeated application is safe.

### Intuition

Tier 1 asks: "What *could* this grammar need, based on its shape?" This is a
coarse, fast approximation that runs on every grammar.

Tier 2 asks: "What *does* this grammar need, based on its predicate semantics?"
This is a precise, formula-driven classification that runs only when predicates
are parsed.

The two tiers are complementary:
- Tier 1 catches structural patterns that predicates alone cannot reveal (e.g.,
  bracket pairing for M4, rule-count ambiguity for M7).
- Tier 2 catches semantic patterns that structure alone cannot reveal (e.g.,
  equality relations for M6, infinite quantification for M2).

## 2. Soundness Lemma

We prove that each of the 6 grammar-structure heuristics in `classify_grammar()`
(Tier 1) is sound: when the heuristic fires, the activated module is genuinely
needed (or at worst redundant, but never harmful).

The argument structure for each heuristic is:
1. Identify the grammar pattern detected.
2. Show that the pattern implies a language-theoretic property.
3. Show that the property requires the activated module's capabilities.

### Heuristic H2: M2 (Buchi) -- Recursive Categories

**Pattern detected**: A category C has C ∈ refs(C), i.e. C directly references
itself in at least one production rule. More generally, C →* C in the category
reference graph (transitive closure contains a cycle through C).

**Soundness argument**.

Let C be a recursive category with C →* C. Consider the derivation trees of
strings generated by C. Because C can expand to a right-hand side containing C,
derivation can recurse unboundedly:

  C ⟹ α₁ C β₁ ⟹ α₁ α₂ C β₂ β₁ ⟹ ···

This generates strings of the form α₁ α₂ ··· αₖ γ βₖ ··· β₂ β₁ for arbitrary
k, where γ is a terminal derivation of C's non-recursive alternative (if one
exists).

The key observation is that the *set* of strings derivable from C is a
context-free language. If we attach predicates to these productions, the
predicate language must constrain an unbounded repetition. Checking that a
predicate holds *for all* derivation depths requires reasoning about infinite
behaviors --- specifically, the ability to express "for all n, the n-fold
unrolling satisfies property P." This is precisely the domain of omega-regular
analysis.

Formally, let L(C) be the language of terminal strings derivable from C. The
pumping lemma for context-free languages guarantees that L(C) contains strings
of unbounded length. Any predicate that constrains the *structure* of these
unbounded derivations (rather than merely their finite prefixes) requires
acceptance conditions over infinite words (Buchi acceptance). Hence M2 is
needed.

**Note on conservatism**: Not every recursive category has predicates that
require omega-regular analysis. A simple recursive arithmetic expression grammar
(Expr → Expr "+" Expr) may only need finite analysis. The heuristic fires
conservatively; M2 activation is harmless when unnecessary because the Buchi
module produces trivially-true results for finite-language inputs.  ∎

### Heuristic H3: M3 (AWA) -- Multi-Branch Rules

**Pattern detected**: At least one production rule has ≥ 3 non-terminal children
in its right-hand side (i.e. `nt_count >= 3`).

**Soundness argument**.

A rule with k ≥ 3 non-terminal children

  R → N₁ t₁ N₂ t₂ N₃ ··· tₖ₋₁ Nₖ

generates parse trees where R has k subtrees, one per non-terminal. Any
predicate that must hold *for all* children simultaneously requires universal
branching --- the ability to verify a property along every branch of a k-way
fork.

Alternating automata (AWA) generalize nondeterministic automata by allowing
transitions to specify both existential and universal successors. A universal
transition

  δ(q, a) = q₁ ∧ q₂ ∧ q₃

accepts iff *all* of q₁, q₂, q₃ accept in their respective subtrees. This is
exactly the acceptance mode needed when a predicate asserts a conjunction of
properties across k ≥ 3 children.

For k = 2 (binary rules), standard product constructions on NFAs suffice.
For k ≥ 3, the product state space grows as |Q|^k, making direct NFA
construction exponential. AWA provide a polynomial representation via
alternation (Chandra, Kozen, & Stockmeyer, 1981).

Therefore, when `nt_count >= 3`, the grammar contains rules whose predicate
analysis may require universal branching, and M3 activation is sound.  ∎

### Heuristic H4: M4 (VPA) -- Paired Delimiters

**Pattern detected**: The terminal symbol set contains at least one "call"
symbol from {"(", "{", "[", "begin", "do"} and at least one matching "return"
symbol from {")", "}", "]", "end", "done"}.

**Soundness argument**.

Alur and Madhusudan (2004) introduced *visibly pushdown languages* (VPL), a
subclass of context-free languages where the input alphabet is partitioned into
three sets:

  Σ = Σ_call ∪ Σ_return ∪ Σ_internal

A visibly pushdown automaton (VPA) pushes on call symbols, pops on return
symbols, and performs internal moves on internal symbols. The crucial property is
that the *stack height at each position is determined by the input word alone*,
not by the automaton's state. This makes VPL closed under all Boolean
operations (unlike general CFLs) and decidable for inclusion and equivalence.

When a grammar contains paired delimiters, the language of valid parse trees has
inherent nested structure: every "(" must match a ")", every "{" must match a
"}", etc. Any predicate that constrains well-nesting (e.g. "every opened scope
is properly closed," "variables declared in an inner scope do not escape")
requires pushdown analysis. The VPA framework provides this analysis with the
added benefit of Boolean closure, enabling composition of multiple nesting
predicates.

The algebraic justification is that well-nested languages have syntactic monoids
in the DA pseudovariety (unambiguous polynomial closure), which corresponds to
M4 in the Eilenberg variety-module correspondence table.

The heuristic checks for the *presence* of paired delimiters, not for
well-nesting. A grammar could have "(" and ")" as terminals without requiring
them to be paired. This is an intentional over-approximation: activating M4 when
it is not needed is harmless, but failing to activate it when nesting predicates
are present would be unsound.  ∎

### Heuristic H5: M5 (Parity Tree) -- Recursive Branching

**Pattern detected**: Both conditions hold simultaneously:
1. There exists a recursive category (C →* C in the reference graph).
2. There exists a rule with ≥ 3 non-terminal children.

**Soundness argument**.

Neither recursion alone (H2) nor branching alone (H3) implies the need for
parity tree automata. Their *conjunction* does: a recursive category with
branching children generates *ranked trees* of unbounded depth.

Formally, let C be a recursive category and R a rule with k ≥ 3 non-terminal
children. The derivation trees rooted at C form an infinite set of finite ranked
trees. If the category reference graph connects C to R (directly or
transitively), then unrolling the recursion produces trees where:
- Each internal node has rank ≤ k (determined by the rule used).
- The tree depth is unbounded (due to recursion).

Parity tree automata (Emerson & Jutla, 1991) are the canonical acceptors for
properties of infinite ranked trees, with parity acceptance generalizing both
Buchi (eventually always) and Rabin (pairs of infinitely-often) conditions. The
mu-calculus (Kozen, 1983) provides the logical characterization: a property is
parity-tree-recognizable iff it is mu-calculus definable.

For PraTTaIL, the connection is: a predicate that constrains the *global*
structure of a recursively-generated tree (not just individual paths through it)
requires tree-level acceptance. M5 provides the parity tree automaton
infrastructure for such analysis.

The algebraic counterpart is the pseudovariety 𝐁𝐕 (block groups), which
characterizes tree-recognizable languages in the Eilenberg correspondence
(Walukiewicz, 2002).  ∎

### Heuristic H6: M6 (Register) -- Binder Items

**Pattern detected**: At least one rule contains a `SyntaxItemSpec::Binder`
item, indicating a grammar construct that introduces a variable binding.

**Soundness argument**.

Binder items introduce *fresh variable scopes*: a binder declares a name that
is visible within a delimited region of the parse tree and must be tracked for:
1. **Scope correctness**: The bound variable is not referenced outside its scope.
2. **Freshness**: The bound name does not collide with other bound names in
   enclosing scopes (alpha-equivalence).
3. **Data comparison**: Predicates may test equality between bound variables
   and other values.

These requirements go beyond regular languages. A language of well-scoped
variable bindings is not regular --- it requires comparing data values
(variable names) across arbitrary distances in the input. This is precisely the
domain of *register automata* (Kaminski & Francez, 1994), which extend finite
automata with a finite set of registers that can store data values and test
incoming symbols against stored values.

The algebraic framework for data languages uses orbit-finite nominal sets
(Bojańczyk, Klin, & Lasota, 2014). The syntactic monoid analogue is an
orbit-finite nominal monoid, which lies outside the classical Eilenberg
hierarchy (hence the "---" entry in the variety-module correspondence table).

Bojańczyk (2019) showed that register automata are equivalent to orbit-finite
nominal automata, establishing the theoretical basis for using M6 to analyze
binding structure.

Therefore, the presence of binder items soundly triggers M6 activation.  ∎

### Heuristic H7: M7 (Probabilistic) -- Rule Ambiguity

**Pattern detected**: Some category C has ≥ 3 production rules:
|{R : R ∈ rules(C)}| ≥ 3.

**Soundness argument**.

When a category has multiple production rules, an input string may be derivable
via more than one rule, creating *parse ambiguity*. For 2 rules, the ambiguity
is binary and can typically be resolved by a single discriminating token
(deterministic choice). For k ≥ 3 rules, the disambiguation becomes a
k-way classification problem.

Probabilistic automata (Rabin, 1963) and stochastic context-free grammars
(Booth & Thompson, 1973) assign weights (probabilities) to production rules,
enabling maximum-likelihood disambiguation:

  argmax_{R ∈ rules(C)} P(R | input prefix)

The threshold k ≥ 3 is motivated by the observation that binary ambiguity is
always resolvable by FIRST set analysis (the standard LL(1)/LR(1) technique),
but ternary and higher ambiguity may have overlapping FIRST sets that require
statistical ranking. Specifically, if three rules R₁, R₂, R₃ have

  FIRST(R₁) ∩ FIRST(R₂) ∩ FIRST(R₃) ≠ ∅

then no single lookahead token can deterministically select among them, and
probabilistic disambiguation becomes necessary.

PraTTaIL's WFST infrastructure provides the weight framework (semirings,
Viterbi decoding, forward-backward computation). M7 connects the dispatch to
this infrastructure, enabling probability-weighted rule selection.

**Conservatism note**: Not every 3-rule category is ambiguous --- the rules
may have disjoint FIRST sets. The heuristic fires conservatively. M7 activation
when not needed merely causes the probabilistic module to compute trivial
uniform weights, which has no effect on the dispatch outcome.  ∎

## 3. Completeness Discussion

### The Asymmetry Principle

The two-tier dispatch architecture is intentionally *asymmetric* with respect to
errors:

| Error Type | Definition | Consequence | Acceptable? |
|------------|-----------|-------------|-------------|
| **False positive** | σ activates Mᵢ but Mᵢ is not needed | Mᵢ runs and produces trivially-true results; wasted work but correct output | Yes |
| **False negative** | σ does not activate Mᵢ but Mᵢ is needed | Mᵢ does not run; its analysis is silently skipped; potential unsoundness | **No** |

This asymmetry arises because PraTTaIL uses module outputs as *preconditions*
for correctness. If M4 (VPA) is needed to verify well-nesting but is not
activated, the pipeline silently skips the well-nesting check, potentially
accepting an ill-formed program. Conversely, if M4 runs unnecessarily, it
verifies a vacuously-true property, which is harmless.

### Formal Statement

**Theorem 3.1** (Soundness -- No False Negatives). For any grammar G:

  need(G) ⊆ σ_final = σ_β ∪ σ_α

*Proof*. We argue each inclusion separately.

*Case 1*: Predicates are available (σ_α is computed).
By Lemma 1.1 of `variety-classification.md`, for every predicate φ:

  { Mᵢ : M(⟦φ⟧) ∈ 𝐕ᵢ } ⊆ extract_features(φ, Γ).signature

Taking the union over all predicates:

  need_pred(G) ⊆ σ_α

The grammar-structure needs (recursion, branching, delimiters, binders, rule
counts) are captured by σ_β per the soundness arguments in Section 2. Therefore:

  need(G) = need_pred(G) ∪ need_struct(G) ⊆ σ_α ∪ σ_β = σ_final  ∎

*Case 2*: Predicates are not yet available (σ_α = ∅).
In this case σ_final = σ_β. The grammar-structure heuristics capture
need_struct(G) as argued above. For need_pred(G), we rely on the fact that when
predicates are unavailable, no predicate-dependent analysis can run regardless
of module activation. Thus no false negative arises from missing σ_α: the
absence of predicate data means predicate-driven modules have no work to do
even if activated.  ∎

### Why Completeness (No False Positives) Is Not Required

**Claim 3.2**. Eliminating all false positives is neither feasible nor
desirable.

*Infeasibility*. Computing need(G) exactly would require:
1. Parsing all predicate formulas (requires the predicated-types pipeline).
2. Computing syntactic monoids for predicate languages (EXPSPACE-hard in
   general; Kozen, 1977).
3. Checking monoid membership in each pseudovariety (polynomial per variety,
   but requires step 2).

Step 2 is the bottleneck: syntactic monoid computation is exponential and
dominates any savings from skipping unnecessary modules.

*Undesirability*. Even if feasible, exact computation would eliminate the
ability to detect modules needed for *future* predicate additions. The
grammar-structure heuristics serve as a "forward-looking" approximation: they
activate modules that the grammar's *structure* suggests *may* be needed, even
before specific predicates are written. This supports incremental grammar
development.

### Quantitative Bounds on Over-Approximation

In practice, the over-approximation is modest. For a grammar with n categories,
r total rules, and t terminal symbols:

- H2 fires iff the category reference graph has a cycle: O(n + edges) to detect.
- H3 fires iff any rule has ≥ 3 non-terminal children: O(r) to detect.
- H4 fires iff the terminal set intersects both call and return symbol sets:
  O(t) to detect.
- H5 fires iff H2 ∧ H3: no additional cost.
- H6 fires iff any rule has a binder item: O(r) to detect.
- H7 fires iff any category has ≥ 3 rules: O(r) to detect.

The heuristics are independent (each can fire or not independently of the
others), so the worst-case over-approximation activates at most 6 additional
modules beyond the base {M1, M10}. The expected over-approximation for typical
programming language grammars is 1--2 extra modules, because:
- Most grammars have recursion (H2 fires) and delimiters (H4 fires).
- Few grammars have 3-way branching (H3 typically does not fire) or binders
  (H6 fires only for lambda-calculus-style languages).

## 5. Monotonicity and Composition Properties

**Lemma 5.1** (Monotonicity of σ_β). Adding rules to a grammar can only set
additional bits in σ_β, never clear them.

*Proof*. Each heuristic Hᵢ is a monotone predicate over the grammar's rule set:
- H2 checks for cycles in the category reference graph. Adding rules can only
  add edges, which can only create new cycles (never remove existing ones).
- H3 checks for ≥ 3 non-terminal children. Adding rules can only introduce new
  high-arity rules.
- H4 checks for terminal symbol pairs. Adding rules can only enlarge the
  terminal set.
- H5 = H2 ∧ H3, conjunction of monotone predicates is monotone.
- H6 checks for binder presence. Adding rules can only introduce new binders.
- H7 checks for ≥ 3 rules per category. Adding rules can only increase counts.

Since σ_β = ⋃ᵢ (Hᵢ fires → {Mⱼ}), and each Hᵢ is monotone, σ_β is monotone
in the grammar's rule set.  ∎

**Lemma 5.2** (Monotonicity of σ_α). Adding predicates to a grammar can only
set additional bits in σ_α, never clear them.

*Proof*. σ_α is defined as the union of per-predicate signatures. Union is
monotone: adding a new predicate φ_{n+1} gives σ'_α = σ_α ∪ sig(φ_{n+1}) ⊇ σ_α.
Within a single predicate, the extraction walk only calls `sig.set()` (bitwise
OR), never clearing bits.  ∎

**Corollary 5.3** (Compositional Soundness). If G₁ ⊆ G₂ (G₂ extends G₁ with
additional rules and predicates), then σ_final(G₁) ⊆ σ_final(G₂). That is,
extending a grammar never reduces the set of activated modules.
